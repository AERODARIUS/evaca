# STORY-20260311-002: Add scheduler idempotency guard for overlapping runs

## What was implemented
- Added scheduler lease semantics for `checkAlerts` using a Firestore-backed lock (`schedulerLeases/checkAlerts`) with expiry.
- `checkAlerts` now skips execution when an active lease exists and releases the lease at the end of the run.
- Added transactional idempotent notification creation using deterministic dedupe keys based on `alertId` and trigger window context.
- Switched alert evaluation notification creation from unbounded `add` to idempotent create-or-skip semantics.
- Added tests for:
  - dedupe key behavior
  - overlapping run idempotency simulation (single notification across concurrent runs)
  - lease acquire/release overlap behavior

## Notes/limitations
- The scheduler lease is best-effort and time-bound; if a worker crashes, lease expiry allows future runs to recover automatically.
